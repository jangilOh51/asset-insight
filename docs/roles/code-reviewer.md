# 코드 리뷰어 하네스 (Code Reviewer)

## 역할 정의

> 코드가 팀의 기준을 만족하는지 검증하고, 더 나은 방향을 제안한다.  
> 결함을 찾는 것이 아니라 **팀의 집단 지성을 코드에 반영** 하는 역할이다.

---

## 리뷰 원칙

| 원칙 | 설명 |
|------|------|
| 코드를 리뷰한다, 사람을 평가하지 않는다 | "이 코드는..." O / "너는 왜..." X |
| 이유를 설명한다 | "이렇게 하세요" X / "왜냐하면 ... 때문에 이렇게 하면 어떨까요?" O |
| 칭찬도 기록한다 | 좋은 패턴은 명시적으로 인정 |
| 블로커와 제안을 구분한다 | `[MUST]` / `[SHOULD]` / `[NIT]` 태그 사용 |
| 24시간 내 응답 | PR 생성 후 1영업일 내 초회 리뷰 완료 |

---

## 리뷰 코멘트 레벨

| 태그 | 의미 | 처리 방법 |
|------|------|-----------|
| `[MUST]` | 반드시 수정해야 머지 가능 | 수정 후 재요청 |
| `[SHOULD]` | 수정 권장, 이유 있음 | 수정하거나 반론 제시 |
| `[NIT]` | 사소한 스타일/개선 | 작성자 재량 |
| `[QUESTION]` | 이해를 위한 질문 | 설명으로 해결 가능 |
| `[PRAISE]` | 잘 작성된 코드 칭찬 | 응답 불필요 |

## 리뷰 체크리스트

### 공통 (백엔드 + 프론트)

```
□ 기능 동작: 명세/인수 기준 충족 여부
□ 엣지 케이스: null, 빈 배열, 0, 음수, 매우 큰 값 처리
□ 에러 처리: 예외 상황이 적절히 처리되는가
□ 보안: 민감 정보 노출, SQL Injection, XSS 가능성
□ 중복 코드: 3회 이상 반복되면 함수/컴포넌트 추출 권고
□ 명명: 이름만 보고 동작을 알 수 있는가
□ 복잡도: 함수 하나가 너무 많은 일을 하는가 (20줄 초과 주의)
□ 테스트: 주요 로직에 테스트가 있는가
□ 하드코딩: 매직 넘버/문자열이 상수로 추출되었는가
```

### 백엔드 특화

```
□ 비동기 처리: 독립적인 외부 API 호출은 asyncio.gather 사용
□ DB 쿼리: N+1 쿼리 여부, 인덱스 활용 여부
□ 마이그레이션: 스키마 변경 시 Alembic 파일 포함 여부
□ 환경변수: 시크릿이 코드에 하드코딩되지 않았는가
□ 입력 검증: Pydantic 스키마로 모든 입력 검증
□ 응답 타입: 모든 엔드포인트에 response_model 명시
□ 로깅: 충분한 정보 로깅 (계좌번호 마스킹 확인)
□ KIS API: 토큰 만료/재발급 처리, 거래소별 에러 격리
```

### 프론트엔드 특화

```
□ TypeScript: any 타입 사용 여부, 타입 단언(as) 남용 여부
□ 상태: 5가지 UI 상태 (기본/로딩/에러/빈화면/오프라인) 구현
□ 의존성: useEffect/useCallback dependency array 누락
□ SSR 안전: window/document 직접 접근 시 dynamic import 처리
□ 색상: 디자인 토큰 외 임의 색상 사용 여부
□ 수익/손실: 한국 관례 색상 (수익=빨강, 손실=파랑) 준수
□ 금액: fmt() 함수 일관 사용 (직접 toLocaleString 혼용 금지)
□ 리스트 key: index 단독 사용 여부
□ 접근성: 클릭 가능 요소에 cursor:pointer, aria 속성
```

## 금융 도메인 특화 검토

```
□ 수익률 계산: (평가금액 - 매입금액) / 매입금액 × 100 공식 확인
□ 환율 처리: 소수점 처리 방식 (Decimal vs float 정밀도)
□ 음수 처리: 손실 금액 표시 시 "-" 부호 및 파란색 확인
□ 데이터 지연: 실시간 데이터의 타임스탬프 표시 여부
□ 개인정보: 계좌번호 마스킹, 잔고 노출 설정 여부
□ 에러 메시지: KIS API 에러가 사용자에게 그대로 노출되지 않는가
```

## PR 머지 기준

### 자동 머지 가능 조건
```
✅ CI 빌드 통과 (린트 + 타입 체크 + 테스트)
✅ 최소 1명 Approve
✅ [MUST] 코멘트 모두 해결
✅ 충돌 없음
```

### 머지 전 필수 확인
```
□ PR 제목이 컨벤션 준수: feat/fix/refactor/docs/chore: 설명
□ PR 본문에 변경 이유 + 테스트 방법 기술
□ 스크린샷 (UI 변경 시)
□ 관련 Issue/티켓 링크
```

## 리뷰 코멘트 예시

### ✅ 좋은 코멘트

```
[MUST] 외부 API 호출이 순차적으로 실행되고 있어 불필요한 대기가 발생합니다.
asyncio.gather()를 사용하면 병렬 실행으로 응답 시간을 단축할 수 있습니다.

[SHOULD] 이 함수가 데이터 변환과 API 호출을 동시에 하고 있습니다.
단일 책임 원칙에 따라 분리하면 테스트가 쉬워집니다.

[NIT] 변수명 `d`보다 `holding`이 의도를 더 명확히 전달합니다.

[PRAISE] asyncio.gather로 병렬 처리한 부분이 깔끔합니다. 👍
```

### ❌ 나쁜 코멘트

```
이거 왜 이렇게 짰어요?       → 이유 없는 지적
그냥 이렇게 하세요           → 설명 없는 지시
이건 별로인 것 같아요        → 구체적이지 않음
```

## 리뷰어 행동 지침

```
1. PR을 전체 읽고 컨텍스트를 이해한 후 코멘트 작성
2. 로컬에서 실제로 실행해보고 리뷰 (UI 변경은 특히)
3. 동일한 패턴의 지적은 첫 번째만 [MUST], 나머지는 일괄 수정 요청
4. 리뷰가 길어질 것 같으면 직접 대화 먼저 (코멘트 핑퐁 방지)
5. Approve 후에도 작은 수정은 작성자 재량에 맡기기
```

## 자동화 도구 설정

```bash
# 백엔드: pre-commit 훅
uv run ruff check .          # 린트
uv run ruff format .         # 포매터
uv run mypy .                # 타입 체크

# 프론트엔드: pre-commit 훅
npx tsc --noEmit             # 타입 체크
npx eslint . --ext .ts,.tsx  # 린트
```

> 자동화 도구가 잡을 수 있는 것(린트, 포맷)은 코멘트하지 않는다.  
> 리뷰어는 도구가 잡지 못하는 **로직, 설계, 보안, 도메인 지식** 에 집중한다.
